# Video Camera Movement

Use this skill before changing Smart Camera, focus tracking, auto-reframe, crop smoothing, or digital stabilization behavior in Media Processor.

This skill is the **implementation guardrail**. The companion skill
`skills/camera-motion-decisions/SKILL.md` is the **decision source of truth**
(what move belongs on a cut). When the two disagree, fix one of them in the
same PR — never silently diverge.

## Current Baseline

- Production is intentionally back on `0.30.22`.
- Draft `49` on project `10` is the known regression canary for camera-motion changes.
- The `0.30.23` through `0.30.37` camera-motion experiments were rejected for visible shake or fallback regressions. Do not revive them by copying code paths back in.
- Two OpenSpec changes are queued to close known gaps between this guardrail and `0.30.22` code: `camera-motion-priority-and-static-kind` and `camera-motion-filter-chain-cleanup`. Read both before touching Smart Camera, auto-reframe, or vidstab paths.

## Priority Order

1. Explicit user intent.
2. Creative camera movement.
3. Smoothed tracking crop path.
4. Digital stabilization cleanup.
5. Final output crop and encode.

## Rules

- User-selected point tracking, custom ROI, or picked object tracking is the highest framing intent. When explicit tracking is set on a cut, any Smart Camera directive on the same cut is discarded at render time.
- Tracking data is a composition target, not a raw per-frame crop command.
- Smart Camera is a creative movement layer. It may add pan, zoom in, or zoom out only when doing so does not override explicit user tracking.
- Smart Camera `none` (schema v3 `kind="none"`) means no extra AI movement. It must not force static crop, vidstab fallback, post-stabilization, or any other correction.
- Smart Camera directives MUST NOT be generated by deterministic modulo-by-cut-order fallback. When Vision fails or returns ambiguous regions, the directive is `kind="none"`. See `skills/camera-motion-decisions/SKILL.md` principle 4.
- Digital stabilization may only remove high-frequency shake. It must not decide framing or change the long-term pan/zoom path.
- A fallback is acceptable only if it is less distracting than the source and does not create single-frame jumps.
- Prefer no extra correction over a correction that looks mechanically locked, rubber-banded, or nervous.
- The decision of what move belongs on a cut is owned by `skills/camera-motion-decisions/SKILL.md`. This guardrail does not specify decision rules; it only enforces the contracts that make those decisions implementable.

## Implementation Guidance

- Build one authoritative crop path per cut before ffmpeg execution.
- Apply smoothing, dead zones, speed limits, acceleration limits, and drift guards before serializing crop commands.
- Do not layer Smart Camera zoompan on top of explicit tracking crop paths.
- Do not layer emotion zoompan (`_zoompan_filter`) on top of any dynamic crop chain — emotion zoom is only valid on a static aspect crop. The `_cut_segment` `elif` ordering must check for an active reframe chain before adding emotion zoompan.
- Thread the project's `crop_region` static anchor through every crop chain, not only the static aspect crop. Smart Camera, point tracking, custom ROI, and YOLO auto-reframe each render their own crop window — the anchor decision belongs to the project, not to the path that happens to render the cut.
- Do not run vidstab or post-stabilization over a segment that already has intentional dynamic framing unless there is a bounded, measured-safe design.
- Keep `none` as pass-through for the creative movement layer.
- Treat localized adjacent-frame spikes as failures even when whole-cut p95 looks acceptable.
- Constants that govern the *decision* of what move to make (e.g., `ZOOM_IN_AREA_MAX`, `PAN_MIN_MEAN_T_DELTA`, `ZOOM_IN_END_SCALE`) live in `skills/camera-motion-decisions/SKILL.md` Layer 3 — that file owns the rationale. Constants that govern *how a chosen move is rendered* (Kalman, smoothing window, vidstab) belong here.

## Verification Requirements

- Verify with production-like renders, not only unit tests.
- Inspect worker logs to confirm which path ran for each canary cut.
- Measure adjacent-frame step p95, p99, and max for the known problem windows.
- Review the rendered MP4 or contact sheets for single-frame jumps.
- If draft `49` gets worse than `0.30.22`, stop and roll back before trying a broader fix.
